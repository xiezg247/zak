核心业务工具归属 — 三种方案对比

回测·选股·数据查询的工具函数应该放在哪个模块？

A

全部放入 vnpy_ashare 作为 ToolProvider

在 AshareEngine 中直接注册工具函数，统一挂载到 LlmEngine

Pros

  • 工具和业务逻辑在同一模块，修改方便
  • 少一层间接调用，性能较好

Cons

  • vnpy_ashare 本就臃肿（27 文件），会进一步膨胀
  • 工具定义与 GUI / 行情 / 引擎混在一起，职责不清
  • 未来新增更多 AI 能力时 ashare 会变成单体巨石
B

全部作为 Python Skills

所有业务工具注册到 vnpy_skills 框架，Skills 直接实现业务逻辑

Pros

  • AI 工具集中在 skills 体系，职责清晰
  • Skills 框架已经是 LlmEngine 的标准入口

Cons

  • Skills 需要持有 MainEngine / BarStore 等依赖，获取困难
  • 回测引擎、数据查询等重量级逻辑不应在 Skills 中重新实现
  • 如果没有 ashare 的 GUI/引擎运行，Skills 工具无法独立工作
C

混合：Skills 做薄壳适配，ashare 做 Service

Python Skills 封装对 vnpy_ashare 服务的调用，业务逻辑仍留在 ashare

Pros

  • 遵循现有模式（VnpyContextSkill 已这样做）
  • vnpy_ashare 保持纯业务逻辑，不必感知 LLM/Function Calling
  • Skills 做薄壳翻译 — 把内部方法映射为 LLM 友好的工具参数
  • 可以在 ashare 中抽 Service 层，被 GUI 和 Skills 共用
  • 测试友好：Service 可独立测试，Skill 可 mock Service

Cons

  • 多一层间接调用，但影响可忽略（都是内存调用）
  • 需要在 ashare 中设计好 Service 接口，初期多些设计工作
